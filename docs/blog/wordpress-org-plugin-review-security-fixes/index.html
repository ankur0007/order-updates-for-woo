WordPress.org plugin review in 2026: 8 security fixes I had to make

July 2026 · ~9 min read

My plugin got sent back twice by the WordPress.org review team. Not because the features were weak — because of small security details most of us never think about. When I submitted my last plugin back in 2022, the review was much lighter. The bar is a lot higher now.

Important: Don’t lean on the Plugin Check (PCP) tool alone. I ran PCP on my code and it passed clean on most of the points below — the human reviewers still flagged them. PCP is a pattern matcher: great at obvious things it can see, but it can’t judge whether your permission check protects the right record, whether your nonce runs before you read input, or whether a footer line breaks a policy.

Passing Plugin Check is the floor, not the finish line.

1. Use %i for table names in $wpdb->prepare()

prepare() only protects values (the %d, %s you pass in). It does not protect table or column names — so building those into the query string by hand is a risk if any part is dynamic. Use the %i placeholder (WordPress 6.2+) for identifiers:

// Before — table name interpolated into the string
$wpdb->get_results( "SELECT * FROM {$table} WHERE order_id = %d" );

// After — %i for the identifier, %d for the value
$wpdb->get_results(
    $wpdb->prepare( "SELECT * FROM %i WHERE order_id = %d", $table, $order_id )
);

2. Use WP_Filesystem, not raw PHP file functions

Skip fopen / fwrite / move_uploaded_file. Go through WordPress’s filesystem API so your code works on every host — including ones where PHP can’t write directly and needs FTP or SSH credentials:

global $wp_filesystem;
if ( ! function_exists( 'WP_Filesystem' ) ) {
    require_once ABSPATH . 'wp-admin/includes/file.php';
}
WP_Filesystem();

$wp_filesystem->put_contents( $path, $data, FS_CHMOD_FILE ); // write
$wp_filesystem->delete( $dir, true );                        // recursive delete

3. Document every outside service you call

If your plugin sends data anywhere off the user’s site — an avatar service, a form handler, a newsletter API — say so in readme.txt: name the service, what data you send, when, and link its Terms and Privacy pages. This surprised me most: even Gravatar counts, because an email hash leaves the site. Add a clear == External services == section.

4. No “Powered by” credit on anything a customer sees

A branding line on a user-facing surface (like an email footer) isn’t allowed by default. Make it a setting that’s off by default, and only ever show it on emails going to your own staff — never to customers.

5. Check the nonce before you touch $_POST / $_GET

A nonce is a one-time token proving the request came from your form, not a forged one. If the request is forged, you shouldn’t even read its data. Verify first, bail, then read:

// Before — reads input, then checks
$note = sanitize_text_field( $_POST['note'] );
if ( ! wp_verify_nonce( $_POST['_wpnonce'], 'save_note' ) ) return;

// After — checks first, then reads
if ( ! wp_verify_nonce( $_POST['_wpnonce'] ?? '', 'save_note' ) ) return;
$note = sanitize_text_field( wp_unslash( $_POST['note'] ?? '' ) );

6. Prefix every option name — and store it as one full literal

Options share one table across the whole site, so a generic name like settings or api_key can collide with another plugin and overwrite or leak data. Everything must be namespaced.

Using a constant for the key is the right practice — one source of truth, no typos. The trap is assembling the key by concatenation, like PREFIX . 'settings'. Plugin Check reads your code as plain text and can’t compute that join, so it only sees the 'settings' tail and flags it. Keep the constant — just give it the complete literal value:

// Bad — key assembled from a prefix + tail (checker only sees 'settings')
const PREFIX = 'myplugin_';
get_option( self::PREFIX . 'settings' );

// Good — constant holds the full literal in one piece
const SETTINGS_OPTION = 'myplugin_settings';
get_option( self::SETTINGS_OPTION );

7. Check permission per object on every REST endpoint

An endpoint’s permission_callback has to do two things: confirm the request is genuine (nonce), and confirm this user may act on this specific record. Two anti-patterns:

  • is_user_logged_in() — far too loose; any logged-in user passes.
  • current_user_can( 'manage_options' ) — too tight and wrong; that’s admin-only, so it locks out the very managers who should use the plugin.

The right check is capability + object ownership — pass the object’s ID into current_user_can:

public function can_access( WP_REST_Request $request ): bool {
    if ( ! wp_verify_nonce( $request->get_header( 'X-WP-Nonce' ), 'wp_rest' ) ) {
        return false;
    }
    $order_id = (int) $request['order_id'];
    return current_user_can( 'manage_woocommerce' )            // store-wide override
        || current_user_can( 'edit_shop_order', $order_id );   // may edit THIS order
}

Passing $order_id is the key — a user who can edit their orders still can’t touch someone else’s. Route every endpoint through one shared gate so the rule is identical everywhere. Plugin Check never flagged the loose versions — this is exactly the logic an automated tool can’t reason about.

8. Escape output late — with WordPress’s own functions

“Escaping” cleans a value so it can’t inject HTML or scripts when it prints. Two mistakes:

  • Escaping early and trusting it later — always escape at the exact moment you echo.
  • Relying on your own helper — this one tripped me up. My code did clean the value, correctly, through a custom function. But Plugin Check only recognizes a fixed list of built-in functions (esc_html, esc_attr, esc_url, wp_kses_post, sanitize_text_field…). It has no idea your function is safe, so it flags the output anyway — and you can’t teach the WordPress.org checker about custom functions. So the code can be correct and still fail the review.

Call the built-in escaper directly, right where you print:

echo esc_html( $customer_name );
echo '<a href="' . esc_url( $link ) . '">' . esc_html( $label ) . '</a>';
echo wp_kses_post( $rich_note );

Same rule for input: sanitize with core functions (sanitize_text_field, absint, wp_unslash…) at the point you read $_POST, not through a wrapper the checker can’t see. And delete every phpcs:ignore comment — hiding a warning isn’t fixing it.

Fun detail: I set an avatar background with the modern oklch() colour in inline CSS. wp_kses strips oklch() as unsafe, so the colour silently vanished — back to plain hex.

The small things we all forget

  • The Requires Plugins: woocommerce header in the main plugin file, so WordPress knows the dependency.
  • wp_upload_dir() for file paths — never a hard-coded constant like WP_CONTENT_DIR, which breaks on sites that move their uploads folder.
  • Not even one inline <script> in admin output — enqueue it from a real .js file.

The review felt slow while I was in it. But it’s the best free security audit a plugin developer will ever get — and it caught things no automated checker would.
